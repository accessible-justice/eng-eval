# Engineering Evaluation

This is a short take-home exercise to understand how you approach real engineering problems — not whether you can pass a coding quiz.

## What we care about

- Engineering judgment under ambiguity
- Code that's readable, typed, and easy to extend
- Clear communication of decisions and tradeoffs
- Knowing what to leave out

## Time commitment

**Plan for 2–4 hours.** A focused, well-reasoned partial solution beats an exhaustive one with poor judgment calls.

Completeness is not the primary signal. **Judgment is.**

## The setup

This repo has a minimal TypeScript + Express skeleton — a basic server, one stub route, a shared type. It's intentionally sparse.

You're free to:
- Restructure the code if you think it helps
- Add dependencies (just justify them briefly)
- Improve things you notice along the way — note what and why

## Tasks

See the [open issues](../../issues). Each is self-contained. **Prioritize Issue 1**, then pick from the rest based on time and interest. You don't need to complete all of them.

## Submission

1. Fork this repo
2. Create a branch: `eval/<your-name>`
3. Open a pull request back to this repo
4. In your PR description, include:
   - What you built and key decisions you made
   - What you'd do differently with more time
   - Assumptions you made where things were unclear
   - Any questions you'd have asked if this were a real project

## Notes

- If something is ambiguous, document your assumption and move on — that's what we'd want on the job
- There's no single right answer
- You're welcome to ask clarifying questions as a GitHub comment before you start
